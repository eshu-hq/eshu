# Incident-Routing Loaders Keep Raw SQL, Not Decode-in-Go (#4683)

Status: Accepted.

Related: Contract System v1 (`docs/internal/design/contract-system-v1.md`),
the #4573 payload-usage manifest gate, and
`go/internal/storage/postgres/incident_routing_sql_schema_lockstep_test.go`.

## 1. Context

Two loaders in `go/internal/storage/postgres` read incident-routing fact
payloads with raw `payload->>'field'` SQL instead of the typed
`sdk/go/factschema` decode seam every other reducer-facing loader uses:

- `listAppliedPagerDutyServiceRoutingQuery`
  (`incident_repository_correlation_loader.go`) filters
  `payload->>'resource_class' = 'service'` and orders by
  `payload->>'provider_object_id'`.
- `serviceIncidentEvidenceQuery`
  (`service_incident_evidence_loader.go`) is a multi-CTE chain that JOINs on
  `payload->>'provider_object_id'` and `payload->>'service_id'` across four
  fact kinds.

Because the #4573 payload-usage manifest gate is generated by scanning
reducer `Decode*` call sites, it cannot see these two loaders' field reads at
all — a gap the in-tree comments previously flagged as "conversion tracked in
#4683," implying decode-in-Go was the eventual destination. This issue closes
that question: it is not a deferral, it is a decision.

## 2. Decision

Keep both loaders as raw, hand-written SQL permanently. Do not convert them to
decode-in-Go. The compensating control is
`TestIncidentRoutingSQLProjectedFieldsAreSchemaDeclared`
(`incident_routing_sql_schema_lockstep_test.go`), not a future migration to
the typed seam.

## 3. Rejected alternative: decode-in-Go

Decoding the full payload into a typed `sdk/go/factschema` struct in Go, then
filtering/ordering/joining in application code, was considered and rejected.

## 4. Rationale — the index and the JOIN are load-bearing

`listAppliedPagerDutyServiceRoutingQuery`'s `resource_class = 'service'`
predicate and `provider_object_id` ordering push directly into the partial
expression index `fact_records_incident_routing_applied_service_idx`
(`schema_fact_records_incident_indexes.go`, mirrored in
`migrations/003_fact_records.sql`):

```sql
CREATE INDEX IF NOT EXISTS fact_records_incident_routing_applied_service_idx
    ON fact_records (
        (payload->>'provider_object_id'), stable_fact_key, fact_id ASC, generation_id
    )
    WHERE fact_kind = 'incident_routing.applied_pagerduty_resource'
      AND is_tombstone = FALSE
      AND payload->>'resource_class' = 'service';
```

A partial index only matches a query whose `WHERE` clause implies the index's
own predicate. Decode-in-Go would abandon the `resource_class = 'service'`
predicate (that filtering would move into Go after the fact), which means
Postgres could no longer use this index at all — not "use it less
efficiently," but structurally ineligible, because the query text would no
longer imply the indexed partial condition.

`serviceIncidentEvidenceQuery` is a five-CTE JOIN chain across
`reducer_service_catalog_correlation`, `reducer_incident_repository_correlation`,
`incident.record`, `incident_routing.applied_pagerduty_resource`, and
`incident_routing.observed_pagerduty_service`, each joined on a
`payload->>'...'` expression. Decode-in-Go cannot push a JOIN into SQL; the
equivalent Go code would mean loading four to five fact kinds' full payloads
into memory per query and doing the join, dedup, and fail-closed
repository-uniqueness gate (see the loader's doc comment) by hand in
application code — reimplementing what Postgres already does correctly and
efficiently.

### Performance Evidence

Measured locally with a throwaway `postgres:16` container (`docker run`, no
persistent volume, torn down after the run): `fact_records` seeded with
45,000 rows (40,000 background rows across `file`, `repository`,
`content_entity`, and `reducer_package_ownership_correlation` fact kinds
spread over 10 scopes, plus 5,000
`incident_routing.applied_pagerduty_resource` rows in one scope/generation,
20% of which carry `resource_class = 'service'`), `ANALYZE`d, with only the
one partial index above present.

**Current shape** (`listAppliedPagerDutyServiceRoutingQuery`, indexed
pushdown):

```
Incremental Sort  (cost=23.62..23.66 rows=2 width=156) (actual time=0.064..1.073 rows=1000 loops=1)
  Sort Key: ((payload ->> 'provider_object_id'::text)), fact_id
  Presorted Key: ((payload ->> 'provider_object_id'::text))
  Buffers: shared hit=459
  ->  Index Scan using fact_records_incident_routing_applied_service_idx on fact_records fact
        (cost=0.28..23.61 rows=1 width=156) (actual time=0.013..0.754 rows=1000 loops=1)
        Index Cond: (generation_id = 'scope-0-gen-1'::text)
        Filter: (scope_id = 'scope-0'::text)
        Buffers: shared hit=456
Planning Time: 0.242 ms
Execution Time: 1.121 ms
```

**Decode-in-Go alternative** (same scope/generation, no `resource_class` or
`provider_object_id` pushdown — the shape decode-in-Go would produce, since
filtering/ordering would move to Go):

```
Seq Scan on fact_records fact  (cost=0.00..2340.50 rows=197 width=114) (actual time=9.852..11.205 rows=5000 loops=1)
  Filter: ((NOT is_tombstone) AND (fact_kind = 'incident_routing.applied_pagerduty_resource'::text)
           AND (scope_id = 'scope-0'::text) AND (generation_id = 'scope-0-gen-1'::text))
  Rows Removed by Filter: 40000
  Buffers: shared hit=1553
Planning Time: 1.828 ms
Execution Time: 11.454 ms
```

**No-Regression Evidence:** the current indexed shape is ~10x faster
(1.121ms vs 11.454ms) and touches ~3.4x fewer buffers (459 vs 1553) at this
modest 45k-row scale; the gap widens with corpus size because the Seq Scan
cost grows with total table size while the Index Scan cost stays anchored to
the matching partial-index row count. This is the proof that the index is
load-bearing and that abandoning its predicate is a real regression, not a
theoretical one.

## 5. Why the #4573 gate structurally cannot cover this

The #4573 payload-usage manifest gate is generated by scanning reducer
`Decode*` call sites in the typed factschema seam. These two loaders read
`fact.payload->>'field'` directly in SQL text — there is no `Decode*` call for
the gate to see. This is by design, not an oversight to fix by migrating the
loaders onto the seam: the SQL needs the raw JSONB operators to reach the
index and the JOIN, so moving onto the typed seam would mean decoding first
and losing exactly the pushdown this ADR keeps.

## 6. Compensating governance

`TestIncidentRoutingSQLProjectedFieldsAreSchemaDeclared`
(`go/internal/storage/postgres/incident_routing_sql_schema_lockstep_test.go`)
is the permanent substitute for the #4573 gate's blind spot: it asserts every
incident-*source*-fact payload field either SQL constant reads via
`payload->>'field'` (or `payload->'field'`) is declared as a property in at
least one generated `incident/v1` JSON Schema, so a future contracts change
that drops or renames one of those fields breaks this test instead of
silently reading a missing column. Reducer-*output* correlation fields
(`provider_service_id`, `repository_id`, `provenance_only`, `outcome` — all
read from `reducer_service_catalog_correlation` /
`reducer_incident_repository_correlation`, not a collector-emitted incident
source payload) are intentionally excluded from that assertion set; they are
governed by the reducer's own correlation-builder contract instead.

As part of this issue, the test's asserted field set was completed: it was
missing `service`, the nested object `serviceIncidentEvidenceQuery` reads via
`payload->'service'->>'id'` (with a `service_id` fallback) to resolve an
`incident.record`'s PagerDuty service reference. `service` is declared in
`incident.record.v1.schema.json`; it just was not part of the lockstep
assertion, so a rename of that property would not have been caught. It is now
asserted.

## 7. Consequences

- `listAppliedPagerDutyServiceRoutingQuery` and `serviceIncidentEvidenceQuery`
  stay raw SQL permanently. No follow-up conversion issue exists or is
  needed.
- `sdk/go/factschema/incident/v1/incident_routing.go` and
  `sdk/go/factschema/incident/v1/AGENTS.md` document the source-fact fields
  these loaders read so the schemas keep declaring them, per
  `TestIncidentRoutingSQLProjectedFieldsAreSchemaDeclared`.
- Any future field removal or rename in the incident source schemas that
  would break one of these two SQL constants fails
  `TestIncidentRoutingSQLProjectedFieldsAreSchemaDeclared` at test time,
  before it can reach a running query.
- A genuinely new hot-path query need against these fact kinds should extend
  the SQL (and, if needed, the lockstep test's asserted field set) rather
  than reopen the decode-in-Go question this ADR closes.
