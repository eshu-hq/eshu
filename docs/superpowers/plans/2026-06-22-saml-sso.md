# SAML SSO Implementation Plan

This note records the bounded implementation plan for issue #3458. It keeps the
runtime work focused on SAML service-provider login while preserving Eshu-owned
authorization, browser sessions, and hash-only identity storage.

## Goal

Add the SAML foundation needed for dashboard SSO:

- parse and validate provider metadata before login can use it;
- produce stable service-provider metadata for a configured tenant provider;
- normalize NameID and group claims without storing raw assertions or claims;
- reject replay, missing groups, clock skew, bad issuer, bad audience, and bad
  ACS inputs before creating a browser session;
- feed successful SAML assertions into the existing browser-session path.

## Package Contract

SAML identifies an external human subject. It does not grant Eshu permissions.
Successful assertions must resolve through Eshu tenant memberships, roles,
grants, policy revision, and browser-session storage before any API request is
authorized.

The SAML package may depend on a maintained SAML parser/validator, but it must
not adopt a third-party cookie or session model. Raw SAML assertions, NameID
values, email addresses, group names, metadata URLs, private endpoints, and
provider secrets must stay out of logs, docs, issues, tests, and durable rows.

## Implementation Slices

1. Create `go/internal/samlauth` with package docs, scoped instructions, value
   types, metadata validation, SP metadata rendering, and claim normalization.
2. Add replay and clock-skew tests around assertion/request identifiers using
   hashed keys and deterministic fixture data.
3. Add a query-layer SAML handler with metadata, login, and ACS routes. The
   handler should create the same `__Host-` browser cookies as the existing
   browser-session route after a store resolves the SAML subject to an
   `AuthContext`.
4. Wire the handler through API startup without growing near-limit files.
5. Update OpenAPI and HTTP API docs for the public SAML metadata/login/ACS
   routes, including the fail-closed and no-raw-secret contract.
6. Create follow-up GitHub issues for credential-bearing Okta proof and any
   shared OIDC/SAML provider-mapping work that cannot be safely completed with
   committed synthetic fixtures.

## Review Checklist

- SAML and Okta proof artifacts use only synthetic `example.test` fixtures.
- New public routes are intentionally bounded and documented.
- No route trusts client-provided roles, grants, tenant, or workspace state.
- Replay checks are atomic at the storage boundary before session creation.
- Missing group claims and stale metadata fail closed for new logins.
- Existing shared token, scoped token, and browser-session behavior is unchanged.
- Package docs explain the operational and security invariants.
- Changed Go files stay below the 500-line limit.
