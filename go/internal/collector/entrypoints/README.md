# Collector Entrypoints

## Purpose

`entrypoints` renders the shared startup, claim service, and generic
claim-runtime config files for hosted collector command packages from
`collector_entrypoints.yaml`.

## Ownership boundary

This package owns deterministic generation for collector command boilerplate.
It does not own provider target decoding, credential resolution policy, source
fact contracts, Helm templates, or runtime scheduling.

## Exported surface

See `doc.go` for the godoc contract. The exported surface is `Manifest`,
`EnvSpec`, `SourceSpec`, `ManifestSet`, `LoadManifestFile`, `Generate`, and
`GeneratedFile`.

## Dependencies

The package uses the Go formatter, `text/template`, and `gopkg.in/yaml.v3`.
Generated files depend on `internal/app`, `internal/runtime`,
`internal/storage/postgres`, `internal/telemetry`, `internal/workflow`, and the
provider package named by each manifest entry.

## Telemetry

The generator emits no runtime metrics, spans, or logs. Generated collectors
continue to use the existing hosted runtime health, readiness, metrics,
`/admin/status`, claim, source, and ingestion telemetry.

Collector Observability Evidence: generated service files attach the existing
provider `Tracer` and `Instruments` fields through a generated runtime-signal
stub; no telemetry instruments, labels, or status payloads change.

No-Observability-Change: this package changes source generation only and keeps
collector runtime telemetry on the existing hosted runtime and provider source
signals.

## Gotchas / invariants

Generated file order is stable and byte-for-byte checked by
`scripts/verify-collector-entrypoints-generated.sh`. Provider-specific
`source_config.go` files stay handwritten so secret handling and target
validation remain explicit.

Collector Performance Evidence: `go test ./internal/collector/entrypoints -count=1`
exercises manifest validation and rendering without adding provider calls,
queue operations, graph writes, or database work.

No-Regression Evidence: the `managedHeader` constant was extended to include the
project SPDX header (MIT) alongside the existing "managed by manifest" comment.
The generated output is byte-identical to the SPDX-header-pass output from
`scripts/add-license-header.sh`; the three generated files per collector
(config.go, main.go, service.go) are identical before and after the change
modulo the two-line header addition. No generated code, import graph, or
compiled binary changes.

Collector Deployment Evidence: no Helm, Compose, image, ServiceMonitor, or
binary-install behavior changes are generated by this package.
