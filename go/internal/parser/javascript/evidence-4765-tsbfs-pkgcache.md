# Package-Root TypeScript Public-Surface Cache Evidence (#4765)

Theory (proven pre-implementation): a throwaway tree-sitter parse-count shim
against the react-native-ui corpus measured 415,104 sibling tree-sitter
parses across 274 TypeScript files (~2,256 sibling parses per affected file),
all re-walking the identical barrel re-export closure rooted at each
package's `package.json` `exports`/`types` entry. App repos with no
`package.json` exports/types field triggered zero sibling parses (the BFS
early-returns via `PackagePublicSourcePaths` returning nil), confirming the
cost is isolated to TypeScript library packages with a public-surface barrel,
not app code.

No-Regression Evidence: `cd go && GOCACHE=/private/tmp/claude-501/gocache-4765b go test ./internal/parser/... -count=1` -> 1516 passed, 41 packages (was 1515 before the two new package-surface cache tests were added). `bash scripts/test-verify-golden-corpus-gate.sh` -> pass. `bash scripts/verify-parser-relationship-kit.sh` -> pass.

Equivalence Evidence: `TestDefaultEngineParsePathPackageSurfaceCacheEquivalence` (`go/internal/parser/engine_javascript_package_surface_cache_test.go`) asserts the per-declaration `dead_code_root_kinds` output for a 6-module barrel-reexport TypeScript package fixture is identical to the pre-cache per-file behavior: every re-exported function gets `typescript.public_api_reexport`, every re-exported interface gets `typescript.public_api_type_reference`, and every file-private helper function gets no `dead_code_root_kinds` at all. This is a 0/0 differential at the real `Engine.ParsePath` entry point (not a helper in isolation), proving the cache is output-preserving for the fixture shapes it exercises.

A first version of this test passed even though `computePackageSurfaceFacts`'s `declarationMentions` field unioned duplicate-named declarations incorrectly (it originally overwrote instead of unioning -- see below) and even though the fixture never exercised the cache-fed cross-file imported-type-reference path, because its only `typescript.public_api_type_reference` assertion was satisfied by the same-file local walk (`javaScriptTypeScriptPublicTypeReferences`), not by data coming out of the package-surface cache. Code review caught both gaps (P2). The fixture (`buildBarrelReexportPackageFixtureWithCrossFileTypeReference`) now also covers:

- **Cross-file, cache-fed imported-type-reference resolution**: `module_0.ts`'s own public interface imports and mentions `SharedToken` from a third file (`shared_types.ts`) that the barrel never re-exports directly. Resolving `SharedToken` as public-surface-reachable requires the BFS to consult `module_0.ts`'s *cached* facts and find the mention -- not the local same-file shortcut. Confirmed as a real guard: deleting `declarationMentions` from `computePackageSurfaceFacts` makes this assertion fail (`dead_code_root_kinds = <nil>, want []string` on the `SharedToken` check).
- **TypeScript declaration merging (the union-vs-overwrite bug, P2-1)**: `module_1.ts` declares `MergedOptions` twice (`export interface MergedOptions {...}` appears twice), each block mentioning a different imported type (`MergeTypeA`, `MergeTypeB`) from a different module. `javaScriptTypeScriptPublicDeclarationMentionsByName` originally wrote `mentions[name] = mentioned`, which overwrites on the second sighting of the same declaration name and silently drops the first declaration's mentions. Confirmed as a real guard: reinstating the overwrite (instead of the union fix) makes this assertion fail (`dead_code_root_kinds = <nil>, want []string` on the `MergeTypeA`/`MergeTypeB` check). The fix unions each declaration's mentioned names into the shared per-name set instead of replacing it, restoring the exact semantics of the removed `javaScriptTypeScriptPublicDeclarationNodes` slice-walk this cache replaced.

Performance Evidence: `TestEngineParsePathComputesPackageSurfaceClosureOnceForSharedBarrel` and a temporary instrumented probe run (removed before commit) measured, for an 8-module barrel package (closure size 9: `index.ts` + 8 `module_N.ts` files) dispatched through 8 sequential `Engine.ParsePath` calls: **9 real sibling-file parses observed with the cache** (exactly the closure size, i.e. each closure node parsed once) versus a **would-be uncached count of 72** (`moduleCount * closureSize` = 8 * 9, matching the pre-fix behavior where every `ParsePath` call re-walked and re-parsed the entire closure from scratch). That is an 8x collapse on this fixture size, and the collapse ratio grows with file count per package (`O(files * closure_size)` -> `O(closure_size)`), matching the proven-theory shape (415,104 sibling parses / 274 files ~= 2,256 walks/file on the react-native-ui corpus, where closure sizes are much larger than this 9-node test fixture). Remote controlled before/after (`eshu-remote-validation`, Linux amd64, NornicDB `eshu-nornicdb-main:d97f02c1`, `GOMAXPROCS=16`, `ESHU_PARSE_WORKERS=16`, `react-native-ui` bind-mounted alone so both runs share isolated conditions): the `parse` stage for `react-native-ui` (324 files) went from **8.42s (main)** to **2.79s (this branch)** â€” a **~3x wall-clock reduction in isolation**. The isolated figure is conservative: the ~2,256 redundant sibling parses/file this fix removes cost far more under full-corpus contention (they steal CPU from all 16 parse workers across ~900 concurrent repos, which is where the original 380ms/file / ~123s react-native-ui outlier was observed). Result classification: Wall-clock win, output-preserving (0/0 equivalence including declaration-merging + cross-file cases, verified red-without-fix).

Concurrency Evidence: `TestEngineParsePathConcurrentPackageSurfaceCacheIsRaceSafe` dispatches 12 files of a shared barrel package concurrently through `Engine.ParsePath` (mirroring the collector's real worker-pool dispatch) and asserts the observed sibling-parse count still collapses to the closure size (<= 13) under concurrent first-touch, i.e. concurrent workers coalesce onto one in-flight computation per `(package root, file path)` key instead of racing duplicate parses. `go test -race ./internal/parser/... -count=1` -> 1516 passed, 41 packages, and the concurrency test specifically was run 5x under `-race` with `-count=5`, all green, no data race reported.

No-Observability-Change: this adds no worker, queue domain, metric name, metric label, runtime knob, backend branch, retry loop, graph query, or new graph-write route. Parser performance remains visible through existing file parse timing and reducer projection telemetry; no new `eshu_dp_*` instrument, span, or log field is introduced.
